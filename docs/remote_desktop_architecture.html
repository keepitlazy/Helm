📐 拓扑总览(与原 VR 串流架构的根本区别)
原架构是单机自串流 demo:同一台机器上 Electron 启动 stream_sender.exe,通过 localhost 看自己的桌面,反向只回传 VR 头部姿态。

新架构是两台机器、双向控制: 被控端(Host)把桌面经 WebRTC 视频轨推给 控制端(Controller);控制端把键鼠事件经 RTCDataChannel 回传, 被控端用原生 SendInput 注入。两端通过信令服务器交换 SDP/ICE, 用 STUN/TURN 完成公网穿透。视频轨的编码、拥塞控制(GCC)、NACK 重传、 带宽估计全部交给 Chromium 内置 WebRTC——这是公网场景下不自己造轮子的关键。
🖥️ 控制端 Controller (操作者的机器 · Electron)
① 接收视频轨 + 渲染 ~16ms
RTCPeerConnection.ontrack → <video> / WebGL
WebRTC 协商出的远端视频轨直接挂到 <video> 元素,Chromium 内部完成解码(NVDEC/D3D11VA)与渲染。 画面元素同时是输入采集面:鼠标/键盘事件落在它上面。

🌐 信令 + NAT 穿透层
信令服务器(WebSocket,自建 Node 服务):仅转发 SDP offer/answer 与 ICE candidate,不碰媒体流。
穿透:STUN 获取公网映射地址 → P2P 直连;直连失败时 TURN 中继兜底。ICE 自动择优。
媒体/控制:连接建立后,视频轨(SRTP)与 DataChannel(SCTP/DTLS)走 P2P 或 TURN,信令服务器不再参与。
P2P 直连(打洞成功) ≈ 公网 RTT | TURN 中继 +5~30ms | DTLS-SRTP 默认全程加密

↑ 控制通道反向 ↑
⑤ 接收控制消息 + 注入键鼠 ~1-3ms
RTCDataChannel.onmessage → IPC/管道 → 原生 SendInput()
被控端 Electron 收到归一化坐标后,乘以本机桌面分辨率换算成绝对像素,交给原生注入器用 SendInput 合成鼠标移动/点击/滚轮和键盘事件。这是整条链路里 Web 无法替代、必须保留原生的部分。
SendInput: MOUSEINPUT(绝对坐标) / KEYBDINPUT(扫描码) 补全 InputInjector.cpp
⚡ 可选:原生低延迟旁路 (保留现有 C++ 管线 · 用于 LAN/极低延迟模式)
若要榨干延迟或在 LAN 下完全掌控编码,可启用旁路: 原生 WGC 采集 → D3D11 BGRA→NV12 → MF/NVENC 编 H.264(现有 sender/ 管线原样复用) → 经 RTCDataChannel 发送 H.264 NAL(沿用现有 proto::FrameHeader 分帧) → 控制端用 WebCodecs VideoDecoder 解码 → Canvas/WebGL 渲染。

代价:DataChannel 不含媒体级拥塞控制,需自己做码率自适应与抖动缓冲,公网下不如视频轨稳。 所以主路径用视频轨,旁路作为 LAN 高性能选项,两者通过同一信令/穿透层共存,运行时切换。

💡 几个关键设计决策
1. 视频交给 WebRTC,而不是自己造传输:公网下 GCC 拥塞控制 / NACK / 带宽估计 / FEC 极难自己写好。主路径用视频轨,把这些全部外包给 Chromium。

2. 输入注入必须原生:浏览器/Electron 无法系统级合成键鼠,只能靠原生 SendInput。这是 sender/ 唯一不可替代、必须保留并补全的部分。

3. 坐标归一化在发送端、映射在接收端:控制端只发 (x/w, y/h) 比例,被控端按本机分辨率换算。两端分辨率/缩放不一致也不出错。

4. 信令与媒体分离:信令服务器只做"牵线"(SDP/ICE 交换),连上后媒体走 P2P/TURN,服务器零媒体负载、易横向扩展。

5. 主路径 + 原生旁路双轨:公网用 WebRTC 视频轨(稳);LAN 想要极低延迟时切原生 H.264 + WebCodecs 旁路(快)。共用同一信令层。

6. 安全是显式的待办:本期 Demo 不做配对码/白名单/会话授权;但 DTLS-SRTP 已默认加密传输。生产化时在信令层加配对码 + 被控端弹窗授权即可,架构已预留接入点。
